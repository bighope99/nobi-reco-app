## 1. 前提条件

### 必要なツール

| ツール | 用途 | インストール確認 |
|--------|------|-----------------|
| Claude Code | AI開発アシスタント | `claude --version` |
| Git + git-worktree-runner | ブランチ管理 | `git gtr --help` |
| GitHub CLI | PR 作成 | `gh --version` |
| Node.js 18+ | スクリプト実行 | `node --version` |

### 必要な設定

- [ ] GitHub にログイン済み: `gh auth status`
- [ ] Notion 連携が有効（Claude Code の MCP 設定）
- [ ] リポジトリのルートディレクトリにいること

### Notion チケットの準備

チケットのステータスが **「承認OK」** になっていること。
ステータスの流れ：

```
未着手 → 進行中 → レビュー依頼 → レビュー完了 → 承認依頼 → 承認OK → 進行中（実装開始）
```

## 2. どのフローを使うか

```
承認OKのチケットが複数ある？
    │
    ├── はい → 3. 一括処理フロー（/ticket-team-workflow）
    │          最大3グループを並列処理
    │
    └── いいえ → チケットは1件？
                  │
                  ├── はい → 4. 個別処理フロー
                  │
                  └── チケットなし → Notion でチケットを作成してから
```

**判断基準**:

| 状況 | 推奨フロー |
|------|-----------|
| 承認OKチケットが2件以上 | 一括処理（チーム） |
| 1件だけ修正したい | 個別処理（単独） |
| 緊急のバグ修正 | 個別処理（単独） |
| 大量のチケット消化 | 一括処理（チーム）を複数回実行 |

## 3. 一括処理フロー（チーム）

### クイックスタート

Claude Code で以下を入力：

```
/ticket-team-workflow
```

**これだけで開始できます。** 以降はClaude が自動でチームを編成し、処理を進めます。

### あなたがやること（操作ポイント）

一括処理中にあなたが判断・操作する場面は **4箇所** です：

#### 操作① グルーピング承認

```
Leader: 以下のグルーピングでよいですか？

グループA: /children/list 関連（3件）
  - [チケット1] 一覧の表示崩れ修正
  - [チケット2] ページネーション追加
  - [チケット3] 検索フィルタ改善

グループB: /records/activity 関連（2件）
  - [チケット4] 保存時のバリデーション追加
  - [チケット5] 日付ピッカーの不具合修正

→ このまま進めますか？ 変更がありますか？
```

**あなたの選択肢**:
- `OK` / `進めて` → そのまま実行
- `グループAとBを統合して` → グルーピング変更
- `グループBは後回しで` → 優先度調整

#### 操作② 実装計画の承認（Plan モード）

各 Coder が Plan モードで計画を提示します。Claude Code の UI に表示されます。

```
=== 実装計画 ===
対象チケット: #1, #2, #3
対象ファイル:
  - app/children/list/page.tsx
  - components/children/children-list-table.tsx
  - __tests__/children-list.test.tsx

実装方針:
  1. テスト追加（表示崩れ再現）
  2. レスポンシブ対応修正
  3. ページネーションコンポーネント追加

懸念事項: なし
```

**あなたの操作**: UI 上の **「承認」ボタン** をクリック（または修正指示を入力）

#### 操作③ CodeRabbit で判断が必要なコメント

```
CodeRabbit から以下のコメントがありました：

🟡 ユーザー判断が必要:
1. components/table.tsx:45 - このコンポーネントのAPI設計を変更すべきか？
   選択肢A: 既存APIを維持してラッパーを追加
   選択肢B: 破壊的変更を許容してAPI刷新

→ どちらにしますか？
```

**あなたの選択肢**: `A` / `B` / 具体的な指示

#### 操作④ PR マージ後の確認

```
以下のPRが作成されました：
- PR #123: /children/list 関連の修正
- PR #124: /records/activity 関連の修正

マージ完了後、worktree を削除してよいですか？
```

**あなたの操作**: GitHub で PR をレビュー・マージ → `マージしたよ` と伝える

### 自動で実行されること

| ステップ | 実行者 | 内容 |
|---------|--------|------|
| チケット取得 | PM | Notion から承認OK チケットを取得 |
| グルーピング | PM | パス別に分類・優先度順に並べ替え |
| Worktree 作成 | PM | `git gtr new <branch>` で作業環境を作成 |
| 実装 | Coder (×最大3) | TDD で並列実装 |
| テスト | Coder | RED → GREEN → REFACTOR |
| コミット | Coder | Conventional Commits 形式 |
| レビュー | Coder/Reviewer | `/pr-review` 実行 |
| PR 作成 | Coder | `/create-pr` → CodeRabbit ループ（最大3回） |
| Notion 更新 | PM | ステータスを「レビュー依頼」に更新 |
| Worktree 削除 | PM | マージ確認後に `git gtr rm` |

## 4. 個別処理フロー（単独）

1件のチケットや緊急修正を処理する場合のフローです。

### 手順

#### Step 1: チケット確認

```
/notion-ticket-workflow
```

Claude が Notion からチケット一覧を取得・表示します。処理したいチケットを指定してください。

#### Step 2: Worktree 作成

```
/create-worktree
```

Claude が作業用の worktree を自動作成します。

> **重要**: main ブランチで直接作業しないでください。必ず worktree を使います。

#### Step 3: 実装を依頼

チケット内容を伝えて実装を依頼します。

```
このチケットを実装して：[チケットのタイトルや内容を伝える]
```

Claude は Plan モード → TDD 実装 → コミットまで自動実行します。
**Plan モードの承認** はあなたが行います（操作②と同じ）。

#### Step 4: PR 作成

```
/create-pr
```

PR 作成 → CodeRabbit ループが自動実行されます。

#### Step 5: マージ & 片付け

GitHub で PR を確認・マージした後、Claude に `マージしたよ` と伝えてください。
worktree が削除されます。

## 5. ユーザーの操作ポイント

### まとめ: あなたが操作する場面

| タイミング | 操作内容 | 頻度 |
|-----------|---------|------|
| グルーピング承認 | OK or 変更指示 | 1回/実行 |
| Plan 承認 | UI の承認ボタン | グループ数分 |
| CodeRabbit 判断 | 選択肢から選ぶ | 0〜数回 |
| PR マージ | GitHub でマージ | グループ数分 |
| Worktree 削除確認 | 「マージしたよ」 | 1回/実行 |

### やってはいけないこと

- main ブランチで直接コミット
- worktree を手動で削除（`git gtr rm` を使う）
- CodeRabbit ループ中に手動でプッシュ
- Plan モードの承認をスキップ

## 6. チーム構成と役割

### ロール一覧

| ロール | AIモデル | 人数 | 主な責務 |
|--------|---------|------|---------|
| **Leader** | Opus | 1 | ユーザーとの対話窓口。グルーピング承認、エスカレーション判断 |
| **PM** | Sonnet | 1 | チケット取得、グルーピング、worktree 管理、Coder への指示出し、Notion 更新 |
| **Coder** | Sonnet | 最大3 | 実装、テスト、コミット、PR 作成、CodeRabbit 対応 |
| **Reviewer** | Sonnet | 0〜2 | 大規模変更時のみ。`/pr-review` を複数観点で実行 |

### Reviewer が追加される条件（大規模変更）

以下の **いずれか** に該当するグループには専任 Reviewer が付きます：

| 条件 | 閾値 |
|------|------|
| 変更ファイル数 | 7ファイル以上 |
| 変更行数（テスト除く） | 200行以上 |
| 新規ファイル追加 | 機能開発を含む |
| DBスキーマ変更 | あり |

### エスカレーションの流れ

```
Coder で解決できない
    │
    ▼
PM に報告
    │
    ├── PM が解決 → Coder に差し戻し
    │
    └── PM でも解決できない
            │
            ▼
        Leader に報告 → ユーザーに判断を仰ぐ
```

**エスカレーション条件**:
- スコープが想定以上（10ファイル超）
- 他チケットとの競合・コンフリクト
- 仕様が不明確で判断できない
- 破壊的変更が必要

## 7. 各フェーズの詳細

### Phase 1: 計画

#### 1.1 チケット取得

```bash
npx tsx .claude/skills/notion-ticket-workflow/scripts/query-tickets.ts --status "承認OK"
```

出力例：
```json
[
  {
    "id": "abc123",
    "title": "児童一覧の表示崩れ",
    "path": "/children/list",
    "priority": "High",
    "content": "モバイル表示時にテーブルがはみ出す..."
  }
]
```

#### 1.2 グルーピングルール

| ルール | 説明 |
|--------|------|
| パス別 | 同じ画面（app/xxx/）に関するチケットをまとめる |
| 最大3グループ | 1回の実行で処理するのは3グループまで |
| 優先度順 | Now > Urgent > High > Normal > Low |
| 締切順 | 同一優先度なら締切が近いものを優先 |

### Phase 2: 実装

#### 2.1 TDD サイクル

```
RED（テスト追加・失敗確認）
  │
  ▼
GREEN（最小限の実装でテスト通過）
  │
  ▼
REFACTOR（テスト通過を維持しつつ改善）
  │
  ▼
次の機能へ → RED...
```

#### 2.2 コミットメッセージ規約

```
feat: 児童一覧にページネーションを追加
fix: モバイル表示時のテーブル崩れを修正
refactor: 児童一覧コンポーネントのリファクタリング
docs: 開発ワークフローのドキュメント追加
```

### Phase 3: レビュー & PR

#### 3.1 `/pr-review` で自動チェックされる項目

| カテゴリ | チェック内容 |
|---------|-------------|
| コード品質 | CLAUDE.md のルール準拠 |
| セキュリティ | インジェクション、認証漏れ |
| パフォーマンス | N+1クエリ、不要な再レンダリング |
| テスト | テストカバレッジの妥当性 |
| 型安全性 | TypeScript 型定義の品質 |
| エラー処理 | サイレント失敗、不適切なフォールバック |

#### 3.2 プロジェクト固有のチェック項目

| 項目 | OK | NG |
|------|----|----|
| Supabase インポート | `@/utils/supabase/server` | `@/lib/supabase/server` |
| Next.js params | `Promise<T>` + `await` | 同期アクセス |
| 認証 | `getAuthenticatedUserMetadata()` | `supabase.auth.getSession()` |
| メンション | `replaceChildIdsWithNames` で変換 | `child:childId` のまま表示 |

#### 3.3 CodeRabbit ループ

```
PR プッシュ
  │
  ▼
5分待機（CodeRabbit レビュー待ち）
  │
  ▼
コメント取得・分類
  │
  ├── 自動修正可能 → 修正 → コミット → プッシュ → ループ先頭へ
  │
  ├── ユーザー判断必要 → ユーザーに報告
  │
  └── スキップ（情報のみ / 対応済み）
  │
  ▼
最大3回ループ → 完了
```

#### 3.4 Notion ステータス更新

```bash
npx tsx .claude/skills/notion-ticket-workflow/scripts/update-ticket-status.ts \
  --page-id <TICKET_ID> \
  --status "レビュー依頼" \
  --pr-url <PR_URL>
```

### メインスキル

| コマンド | 説明 | いつ使うか |
|---------|------|-----------|
| `/ticket-team-workflow` | チケット一括処理 | 承認OKチケットが複数あるとき |

| コマンド | 説明 | いつ使うか |

| `/add-review-point` | レビュー項目追加 | 新しいチェック項目が必要なとき |

### スキルファイルの場所


## 9. トラブルシューティング

### チケットが取得できない

| 原因 | 対処 |
|------|------|
| ステータスが「承認OK」でない | Notion でステータスを確認・変更 |
| プロジェクトフィルタが違う | 「のびレコ開発」に設定されているか確認 |
| Notion MCP が未接続 | Claude Code の MCP 設定を確認 |

### CodeRabbit のレビューが来ない

| 原因 | 対処 |
|------|------|
| PR が push されていない | `gh pr view` で確認 |
| CodeRabbit が無効 | リポジトリの `.coderabbit.yaml` を確認 |
| 待機時間不足 | 5分以上待ってからリトライ |

### Worktree のコンフリクト

| 原因 | 対処 |
|------|------|
| main が進んでいる | `git rebase main` してやり直し |
| 他の Coder と競合 | PM → Leader にエスカレーション |

### Plan モードが終わらない

| 原因 | 対処 |
|------|------|
| 承認ボタンが見えない | Claude Code UI をスクロール |
| 計画に問題がある | 修正指示を入力して再計画 |

### エスカレーションされた場合

Claude から判断を求められたら、以下を伝えてください：

- **スコープ**: 「そのまま進めて」or「範囲を絞って」
- **仕様**: 具体的な方針を指示
- **コンフリクト**: どちらを優先するか
- **破壊的変更**: 許容するかしないか
