# Notion チケット管理ワークフロー

のびレコ開発プロジェクトのNotionチケット（db_nr_タスク）を管理するスキル。

## データベース情報

- **データベース**: db_nr_タスク
- **データソースURL**: `collection://2b2092aa-b014-8033-a92f-000bbe0df3cd`
- **データベースURL**: `https://www.notion.so/2b2092aab0148012aeaff214786b8832`
- **プロジェクトフィルタ**: のびレコ開発 (`https://www.notion.so/2b2092aab014800d8beae48db2dae981`)

## ステータスの意味と運用ルール

| ステータス | 意味 | 次のアクション |
|-----------|------|--------------|
| **未着手** | まだ手をつけていない | 作業開始時に「進行中」へ |
| **進行中** | 実装作業中 | 完了したら「レビュー依頼」へ |
| **レビュー依頼** | コードレビュー待ち | レビュー後「レビュー完了」へ |
| **レビュー完了** | レビュー通過 | 「承認依頼」へ |
| **承認依頼** | 修正するかどうかの判断待ち | ユーザーが判断。勝手に承認しない |
| **承認OK** | 実装してよい。**これが実装対象** | 作業開始時に「進行中」へ |
| **承認保留** | 判断保留中 | ユーザーの指示を待つ |
| **質問** | 仕様や実装方法の相談事項 | ユーザーと議論して方針決定 |
| **回答** | 質問への回答済み | 内容確認後、必要なら実装チケットへ |
| **終了** | 完了 | - |
| **解決** | 対応完了 | - |
| **却下** | 対応不要と判断 | - |

### 実装の優先順位

承認OKのチケットを以下の順で対応する:
1. **優先度**で並べる（今すぐ > 急いで > 高め > 通常 > 低い）
2. **パス（対象ページ）が同じもの**をまとめて対応する — 同じページに関するバグや機能変更を一括で処理すると効率的
3. **予定期日**が近いものを優先

## 操作パターン

### 1. チケット一覧の取得（スクリプト推奨）

**方法A: query-tickets スクリプト（推奨）**

Notion APIで**ステータスプロパティを完全一致フィルタ**して一括取得する。本文・コメント・全プロパティを含む。

```bash
# 承認OKチケットを取得（デフォルト）
npx tsx .claude/skills/notion-ticket-workflow/scripts/query-tickets.ts

# ステータスを指定
npx tsx .claude/skills/notion-ticket-workflow/scripts/query-tickets.ts --status "進行中"

# 件数制限
npx tsx .claude/skills/notion-ticket-workflow/scripts/query-tickets.ts --status "承認OK" --limit 10

# JSON出力をファイルに保存
npx tsx .claude/skills/notion-ticket-workflow/scripts/query-tickets.ts > /tmp/tickets.json
```

前提: `.env.local` に `NOTION_TOKEN=secret_xxx` を設定済みであること。

出力JSON:
```json
{
  "status_filter": "承認OK",
  "total": 10,
  "tickets": [
    {
      "id": "page-uuid",
      "url": "https://notion.so/...",
      "properties": { "名前": "...", "ステータス": "承認OK", "パス": "/records/status", ... },
      "content": "ページ本文...",
      "comments": [{ "author": "...", "text": "...", "created_at": "..." }]
    }
  ]
}
```

**方法B: notion-search（MCP、フォールバック用）**

`notion-search` ツールで `data_source_url` を指定して検索する。セマンティック検索のためプロパティ完全一致ではない。

```
ツール: notion-search
query: "承認OK のびレコ開発"（または関連キーワード）
data_source_url: "collection://2b2092aa-b014-8033-a92f-000bbe0df3cd"
```

検索結果から各チケットを `notion-fetch` で詳細取得し、ステータスを確認する。

承認OKのチケットをパス別にグルーピングして表示する。

表示フォーマット:
```
## パス: /records/status
| ID | 名前 | トラッカー | 優先度 |
|----|------|-----------|--------|
| #170 | 一括作成が機能しない | バグ | 通常 |

## パス: （未設定）
| ID | 名前 | トラッカー | 優先度 |
|----|------|-----------|--------|
| #216 | パスワード設定をして次に飛ぶ先が... | バグ | 通常 |
```

### 2. チケットのステータス更新

実装完了後、`notion-update-page` でステータスを更新する。

```
ツール: notion-update-page
page_id: <チケットのID>
properties:
  ステータス: "進行中"  （または適切なステータス）
```

基本フロー:
- 作業開始 → 「進行中」に更新
- 実装・テスト完了 → 「レビュー依頼」に更新
- PR作成時 → 「レビュー依頼」に更新 + PRコメントを追加（パターン3参照）
- 承認依頼ステータスは**ユーザーに確認してから**設定する

### 3. PR作成時のチケット更新

PRを作成したとき、対応するチケットのステータスを「レビュー依頼」に更新し、PRリンクをコメントとして追加する。

**ステップ1: ステータスを「レビュー依頼」に更新**

```yaml
ツール: notion-update-page
page_id: <チケットのID>
properties:
  ステータス: "レビュー依頼"
```

**ステップ2: PRリンクをコメントとして追加**

```yaml
ツール: notion-create-comment
page_id: <チケットのID>
content: "PR #<PR番号> を作成しました\n<PR URL>"
```

### 4. 新規バグチケットの作成

作業中にバグを発見した場合:

```
ツール: notion-create-pages
parent: { data_source_id: "2b2092aa-b014-8033-a92f-000bbe0df3cd" }
pages: [{
  properties: {
    "名前": "バグの説明",
    "トラッカー": "バグ",
    "ステータス": "承認OK",
    "優先度": "通常",
    "パス": "/affected/path",
    "プロジェクト": ["https://www.notion.so/2b2092aab014800d8beae48db2dae981"]
  },
  content: "バグの詳細説明"
}]
```

デフォルト値:
- **トラッカー**: バグ（タスク分解・計画系はWBS、他メンバーへの情報共有はメモ）
- **ステータス**: 承認OK（すぐ直せるもの）、承認依頼（判断が必要なもの）
- **優先度**: 通常（緊急度に応じて変更）
- **プロジェクト**: のびレコ開発（必ず設定）
- **担当者**: 空欄（指定しない）

### 5. 質問チケットの作成

仕様確認や実装方針の相談が必要な場合:

```
ツール: notion-create-pages
parent: { data_source_id: "2b2092aa-b014-8033-a92f-000bbe0df3cd" }
pages: [{
  properties: {
    "名前": "質問の要約",
    "トラッカー": "メモ",
    "ステータス": "質問",
    "優先度": "通常",
    "パス": "/related/path",
    "プロジェクト": ["https://www.notion.so/2b2092aab014800d8beae48db2dae981"]
  },
  content: "## 質問内容\n詳細な質問\n\n## 背景\nなぜこの質問が必要か"
}]
```

### 6. WBSチケットの作成

要件定義の結果をタスク分解したり、計画に落とし込むときはWBSトラッカーを使う。

### 7. 情報共有（メモ）チケットの作成

他のメンバーに共有すべき情報（仕様の決定事項、技術的な注意点など）がある場合、
トラッカーを「情報共有」にしてチケットを作成する。

### 8. 作業中にバグを見つけたが今は直さない場合

チケットだけ作成してメモに残す。ステータスは承認OKにしておく。
パスを設定しておくと、次にそのパスの作業をする際にまとめて対応できる。

## 注意事項

- **承認依頼のチケットを勝手に承認OKにしない** — ユーザーに判断を仰ぐ
- **質問チケットの回答を勝手に判断しない** — ユーザーと議論する
- チケット作成時は必ず**プロジェクト**にのびレコ開発を設定する
- **パス**はアプリ内のルートパスを設定する（例: `/records/status`）
- 対象バージョンは基本的に「0.1」
